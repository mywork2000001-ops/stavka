# MATH CORE PROMPT — Multi-Sport Value Betting Quantitative Engine

**v2.** Дистиллированная инженерная спецификация, выведенная из `bukmeker.txt` и
расширенная по прямому запросу до мультиспортивной, мультистрановой платформы с
монетизацией купонов и AI-коннектором данных. Область — математическое и
интеграционное ядро (без Node/React/RabbitMQ/K8s инфраструктуры). Этот файл — не
только исходный промт, но и актуальный протокол принятых архитектурных решений:
он обновляется по мере того, как ядро растёт.

## Роль
Ты — квант-разработчик и платформенный инженер, реализующий детерминированное,
воспроизводимое ядро оценки вероятностей исходов **любых видов спорта, любых стран
и лиг**, обнаружения value bets, сборки монетизируемых купонов и подключения
произвольных источников данных через ИИ-маппинг схемы — пригодное для дальнейшей
интеграции в промышленный пайплайн (Data Connectors → Feature Store → Model →
Calibration → Margin Removal → Value Detection → Bankroll/Coupon → Monetization).

## Расширение объёма (v2): мультиспорт, мультистрана, монетизация, AI-коннектор

1. **Универсальность вместо жёсткого перечисления.** «Все страны мира, все клубы»
   технически невозможно и бессмысленно захардкодить — вместо этого сущностная
   модель (`Sport → Country → League → Competitor`) не имеет встроенных
   ограничений на конкретные страны/лиги/клубы/виды спорта. Полнота охвата
   достигается подключением внешних источников данных (см. п.4), а не списком в
   коде. В комплекте — небольшой иллюстративный seed (5 стран, 3 вида спорта,
   6 лиг, 16 участников) для демонстрации архитектуры, не претендующий на полноту.
2. **Мультиспорт через семейства моделей**, а не через переписывание футбольной
   логики: `ScoringModel.POISSON_GOALS` (футбол, хоккей, гандбол — уже
   реализованные Poisson/Dixon-Coles модели), `ScoringModel.POINT_SPREAD`
   (баскетбол, американский футбол — Normal-модель маржи/тотала по CLT),
   `ScoringModel.SET_BASED` (теннис, волейбол — race-to-N-sets по
   отрицательному биномиальному распределению). Каждое семейство выдаёт словарь
   вероятностей исходов, совместимый с тем же движком value-detection/Kelly/coupon.
3. **Монетизация купона** — расчётная, не платёжная: купон с общим
   (совместным) коэффициентом, ставкой, комиссией платформы (% от выплаты) и
   чистой выплатой. Никакой интеграции реальных платежей — это осознанная
   граница объёма (см. ниже).
4. **AI-коннектор данных для произвольного API-ключа.** Пользователь передаёт
   (а) ключ произвольного источника спортивных данных и (б) свой Anthropic API
   ключ. Система реально обращается по HTTP к источнику, берёт один
   пример-запись из ответа и реально вызывает Anthropic API с просьбой
   сопоставить поля этой неизвестной JSON-схемы с каноническими полями
   платформы (match_id, home_team, away_team, odds и т.д.). Дальше маппинг
   применяется ко всем записям. Это заменяет ручное написание парсера под
   каждого провайдера.

## Обязательные математические компоненты (v1, футбольное ядро)

1. **Взвешивание истории**: экспоненциальное затухание `w(t) = e^{-λt}`, λ по умолчанию
   0.005 (полураспад ≈138 дней); скользящее взвешенное среднее и EMA.
2. **Рейтинги команд**:
   - Elo с домашним преимуществом `H`: `E = 1 / (1 + 10^{-(R_home - R_away + H)/400})`.
   - Байесовский рейтинг: гауссово убеждение `θ ~ N(μ0, σ0²)`, апостериорное обновление
     по каждому результату.
   - Poisson strength (атака/защита): `log(λ_home) = μ + home + attack_i - defence_j`,
     подгонка через максимизацию правдоподобия.
3. **Модели голов**: Poisson, бивариатный Poisson (ковариационный параметр λ3),
   Dixon-Coles (коррекция низких счётов через τ(0,0),τ(1,0),τ(0,1)), Negative Binomial
   (overdispersion), Skellam (разность голов), Monte-Carlo симуляция.
4. **Калибровка**: Platt scaling (логистическая регрессия поверх сырых оценок),
   изотоническая регрессия, Expected Calibration Error (ECE), Brier score, Log Loss.
5. **Удаление маржи**: метод Shin (решение по λ через `brentq`) и мультипликативный
   метод как baseline/fallback.
6. **Value Detection**: `EV = p·(odds-1) - (1-p)`, `Value% = p·odds - 1`,
   `Overlay = p_model - p_fair`, `Probability Edge = p_model/p_fair - 1`, bootstrap CI
   вероятности модели.
7. **Bankroll**: (Fractional) Kelly `f* = (p·(b-1) - (1-p)) / (b-1) = EV/(b-1)`, жёсткие
   лимиты (max stake %, max daily exposure).
8. **Coupon Generator**: перебор комбинаций до N событий, ограничение по корреляции
   исходов, максимизация ожидаемого логарифмического роста капитала (Half Kelly на
   совместную вероятность).

## Требования к качеству
- Без look-ahead и survivorship bias в интерфейсах (явные `as_of_date` там, где уместно).
- Каждая функция — чистая, детерминированная, типизированная (type hints), с docstring
  только там, где не очевиден смысл формулы.
- Численная устойчивость: защита от `odds <= 1`, `p` вне `[0,1]`, вырожденных матриц.
- Тестовое покрытие: unit-тесты на каждую математическую формулу с проверкой против
  независимого источника истины (scipy.stats, ручной расчёт, свойства суммы вероятностей).
- Итоговая демонстрация обязана прогонять полный путь: рейтинги → λ голов →
  Dixon-Coles матрица → 1X2 вероятности → Shin fair odds → EV/Value% → Kelly stake →
  генерация купона — на конкретном примере матча с печатью результатов.

## Критические ошибки, которых нужно избежать (см. bukmeker.txt §3.8)
Data/Look-Ahead Leakage, Overfitting, Selection/Survivorship Bias, Ignoring Margin,
Kelly Overbetting, Small Sample Bias, Correlation Risk, Closing Line Bias.

### Ключевые инженерные решения и их обоснование
- **`PoissonStrength.fit`** оценивается через `scipy.optimize.minimize` (L-BFGS-B) с
  ridge-штрафом на attack/defence (0.01·Σθ²) для идентифицируемости параметров —
  без штрафа система вырождена (attack/defence можно сдвинуть на константу).
- **`dixon_coles_matrix`** клиппит отрицательные вероятности после τ-коррекции и
  ренормализует; предупреждение в docstring, что `rho` вне ~[-0.2, 0.2] не гарантирует
  валидности без клиппинга.
- **`bivariate_poisson_matrix`** проверен тестом на то, что при `lambda3=0` совпадает
  с независимым Poisson — это гарантирует, что общий (не частный) случай реализован
  правильно.
- **`shin_margin_removal`** явно бросает `ValueError`, если сумма implied-вероятностей
  ≤ 1 (нет маржи для снятия) — вместо тихого некорректного результата от `brentq`.
- **`outcome_probs_from_matrix`**: матрица индексируется `[home_goals, away_goals]`;
  home_win — нижний треугольник (`tril`, k=-1), away_win — верхний (`triu`, k=1).
  Перепутать их — самый вероятный источник silent bug, поэтому есть отдельный тест
  на вырожденной матрице с массой в одной ячейке.
- **`weighted_rolling_mean`** принимает `as_of_date` и фильтрует `df[date_col] < as_of_date`
  до агрегации — единственная точка, где предотвращается look-ahead в features.
- **`generate_coupons`** использует эвристику корреляции (`pairwise_correlation`):
  один матч → 1.0, общая команда → 0.6, одна лига → 0.25, иначе 0.0; порог по
  умолчанию 0.3 из документа. Это осознанное упрощение — реальная система должна
  заменить эвристику на исторически откалиброванную корреляционную матрицу исходов.
- **CLI**: `bukmeker demo` (после `pip install -e .`) и `python demo.py` (без
  установки) запускают идентичный сквозной прогон на synthetic-данных.
- **CI**: `ruff check .` + `pytest tests/ -q` на Python 3.11 и 3.12 при push/PR в `main`.

### Ключевые инженерные решения v2 (мультиспорт / монетизация / AI-коннектор)
- **`entities.py`**: единая иерархия `Sport → Country → League → Competitor` вместо
  отдельных моделей `Team`/`Player` — теннисный игрок и футбольный клуб оба являются
  `Competitor` с полем `kind` (`club`/`player`/`national_team`), что позволяет
  value-betting движку (`ValueBetCandidate.team_ids`) работать одинаково для
  командных и индивидуальных видов спорта без изменения кода купонов.
- **`ScoringModel`** — это перечисление, привязанное к `Sport`, а не к лиге или
  стране: любая новая лига любой страны автоматически "знает", какое семейство
  моделей вероятностей к ней применимо, без ручной настройки.
- **`basketball.predict_moneyline`/`predict_total`** используют Normal-приближение
  маржи/тотала (обоснование — CLT при 80–120+ независимых владениях мячом), а не
  Poisson по очкам, как в футболе — попытка применить Poisson к баскетбольным
  очкам даёт грубо некалиброванные хвосты распределения.
- **`tennis.race_to_win_prob`** реализует точную формулу "гонки до W побед"
  (`sum_k C(W-1+k, k) p^W (1-p)^k`), проверенную тестом против известной
  закрытой формулы для best-of-3 (`p²(3-2p)`) — сеты считаются iid Bernoulli,
  что игнорирует альтернацию подачи и внутригеймовую динамику (осознанное
  упрощение, см. docstring).
- **`monetization.py`** сознательно отделён от `coupon.py`: генерация купона
  (корреляция, EV, Kelly) — чистая математика; `build_coupon_report` — бизнес-
  логика ценообразования (комиссия платформы как % от gross payout, по образцу
  индустрии coupon/tips-маркетплейсов). Разделение позволяет менять модель
  монетизации, не трогая математику отбора ставок.
- **AI-коннектор (`connectors/`)** сделан из четырёх независимых, инжектируемых
  по конструктору частей: `RawDataSource` (реальный HTTP, ключ в заголовке ИЛИ
  query-параметре — оба варианта явно поддержаны), `ClaudeFieldMapper` (реальный
  вызов Anthropic Messages API; принимает `client=` для внедрения фейка в тестах
  без сетевых вызовов и трат), `find_record_list`/`get_by_path` (эвристический
  BFS-поиск массива записей в незнакомой JSON-структуре + разрешение dotted-path
  с поддержкой индексов списков), `AIDataConnector` (склеивает всё в
  `fetch_and_normalize()`). Такое разделение — единственный способ протестировать
  сетевой/AI-код детерминированно (96 тестов проходят без единого реального
  HTTP- или API-запроса).
- **`bukmeker connector` CLI** явно отказывается работать без реальных ключей
  (печатает usage и возвращает код 1) — никакого синтетического fallback,
  который мог бы создать иллюзию, что данные реальны, когда это не так.

### Ключевые инженерные решения v3 (веб-дашборд)
- Изначальный запрос был "открыть как приложение"; уточнение с пользователем
  показало, что имелся в виду не установочный `.exe`/desktop-окно, а
  интерактивный интерфейс в браузере — выбран **Streamlit** как самый быстрый
  путь к рабочему UI без Node/React, при этом визуально ближе всего к
  дашборду, описанному в исходном `bukmeker.txt` §1.2 ("Frontend — Дашборды").
- **`bukmeker/dashboard.py` не содержит своей математики** — каждая вкладка
  только вызывает уже протестированные функции (`margin`, `models`,
  `value_betting`, `sports`, `coupon`, `monetization`, `entities`) и рендерит
  результат. Это гарантирует, что дашборд не может разойтись с CLI/API по
  цифрам — числа в разделе "Купон и монетизация" дашборда идентичны
  `bukmeker demo` для тех же входных данных (проверено тестом).
- **`bukmeker dashboard` CLI** — не встроенный сервер, а `subprocess.call(["streamlit",
  "run", ...])`; если `streamlit` не установлен (`shutil.which` не находит
  бинарник), печатает инструкцию по установке extras вместо трейсбека.
- **Границы ввода на UI явно обработаны, а не понадеялись на "пользователь не
  ошибётся"**: коэффициенты без букмекерской маржи (`shin_margin_removal`
  бросает `ValueError`) перехватываются на дашборде и показываются как
  предупреждение (`st.warning`), а не роняют страницу — реальный сценарий,
  так как пользователь свободно двигает числовые поля.
- **Тестирование без браузера**: `streamlit.testing.v1.AppTest` прогоняет
  `dashboard_app.py` headless, проверяет отсутствие исключений на старте,
  переключение на вкладку купонов, клик по кнопке генерации и совпадение
  цифр монетизации с ручным расчётом (570/28.50/541.50) — тот же уровень
  проверки, что и у чисто библиотечного кода, без ручного клика для каждой
  правки.

### Ключевые инженерные решения v4 (пароль на дашборд)
- **Контекст решения.** Пользователь попросил "создать профессиональный промт
  и выполнить" без указания конкретной фичи и отклонил уточняющий вопрос —
  решение о содержании этого инкремента принято автономно, по следующему
  критерию: что превращает уже работающий дашборд в более профессиональный
  инструмент, полностью реализуемо и проверяемо в этой среде (без Docker,
  без реальных внешних API-ключей, без ручного клика в браузере), и не
  требует необратимых или дорогостоящих действий. Выбор: `bukmeker
  dashboard` по умолчанию не имел вообще никакой защиты доступа — любой, у
  кого есть сетевой доступ к порту 8501, мог открыть банкролл/купоны.
- **`bukmeker/auth.py`**: один общий пароль на всё приложение через
  `BUKMEKER_DASHBOARD_PASSWORD` (env var), а не полноценная многопользовательская
  авторизация — по объёму сопоставимо с остальными сознательными упрощениями
  проекта (например, эвристика корреляции в `coupon.py`). Полноценные аккаунты
  с раздельным банкроллом потребовали бы БД, что выходит за рамки текущего
  объёма (Python-библиотека + дашборд без backend).
- **Дашборд по умолчанию открыт, а не закрыт.** Если `BUKMEKER_DASHBOARD_PASSWORD`
  не задан, `require_password()` пропускает пользователя дальше, но печатает
  видимое предупреждение (`st.warning`) — сохранён принцип "работает из коробки
  без дополнительной настройки" (как и `bukmeker demo`), при этом небезопасное
  состояние не скрыто от пользователя молча.
- **`st.secrets` сознательно не используется** — только переменная окружения.
  API `st.secrets` версионно-хрупкий (по-разному ведёт себя без `secrets.toml`
  в разных версиях Streamlit), а env var — простой, универсальный и достаточный
  для однопользовательского общего пароля.
- **Проверено headless-тестами** (`tests/test_auth.py`, 3 теста): (1) без пароля
  дашборд открывается сразу с предупреждением на экране, вкладки видны; (2) с
  паролем — вкладки скрыты до входа, неверный пароль показывает ошибку и не
  пускает, верный — открывает; (3) состояние входа переживает последующие
  rerun'ы (например, клик по любому виджету на вкладке), а не спрашивает пароль
  заново на каждое действие.

### Ключевые инженерные решения v5 (профессиональный фронтенд: понятное меню)
- **Контекст решения.** Пользователь сообщил: "ничего в меню непонятного" —
  прямая обратная связь по UX, что плоский ряд вкладок без пояснений не
  самообъясним. Это конкретная жалоба, в отличие от предыдущего безадресного
  "создай промт и выполни" — реализовано именно то, на что указали.
- **`st.tabs` → `st.navigation` + `st.Page`**: плоский список вкладок заменён
  на настоящее меню в сайдбаре с иконками и заголовками (⚽/🏀/🎾/🎫/🌍/ℹ️/❓) —
  это нативный Streamlit-паттерн для многостраничных приложений, читается как
  меню профессионального приложения, а не как ряд кнопок.
- **Добавлена страница "❓ Справка" как страница по умолчанию** (открывается
  первой): что делает приложение, что означает каждый пункт меню, и таблица-
  глоссарий терминов (p_model, odds, EV, Value%, overround, fair probability,
  Kelly stake, λ, ρ, комиссия платформы) простым языком — прямой ответ на
  "непонятно, что тут вообще происходит".
- **`help=` подсказки на каждом техническом поле ввода** (все ползунки/поля
  коэффициентов на всех страницах) — не заставляют пользователя уходить в
  документацию за расшифровкой λ_home/ρ/Kelly fraction, значок `(?)` объясняет
  на месте. Дополнительно — разворачиваемые пояснения (`st.expander`) под
  результатами ("Что означают эти столбцы?", "Как заполнять таблицу ставок?").
- **Файловая структура многостраничности (`dashboard_pages/`), а не функции
  напрямую в `st.Page`.** `AppTest.switch_page()` (headless-тестирование)
  умеет адресовать страницы только по пути к файлу, не по объекту функции —
  поэтому каждая страница вынесена в собственный тонкий файл-обёртку
  (`dashboard_pages/football.py` и т.д.), который просто вызывает
  соответствующую `render_*_page()` из `dashboard.py`. Без этого пришлось бы
  либо не тестировать переключение между страницами вообще, либо кликать
  вручную в браузере при каждой правке.
- **Пути к страницам — абсолютные** (`Path(__file__).resolve().parent / "dashboard_pages" / ...`),
  а не относительные строки — иначе поведение отличалось бы в зависимости от
  того, что именно передано в `streamlit run` (`bukmeker/dashboard.py` из
  продакшен-CLI vs. `dashboard_app.py`-обёртка из корня для локального
  запуска без установки пакета); абсолютный путь работает одинаково в обоих
  случаях (проверено headless-тестом на оба входа).
- **Тестовая цель для `AppTest` перенесена** с `dashboard_app.py` на
  `bukmeker/dashboard.py` напрямую — это именно то, что реально запускает
  `bukmeker dashboard` в продакшене; `dashboard_app.py` остаётся тонкой
  удобной обёрткой для разработки без установки пакета, но сам не является
  источником истины для тестов.
- **Проверено headless-тестами** (`tests/test_dashboard.py`, `test_auth.py`):
  дефолтная страница — справка; каждая из 7 страниц доступна через

### Ключевые инженерные решения v6 (реальный список стран + синхронизация через API-ключ)
- **Контекст решения.** Пользователь попросил "синхронизировать все команды/клубы
  со всемирной системой для всех видов спорта ставок... чтобы в системе были
  все команды", и отдельно — "промт для синхронизации всей игровой системы с
  помощью API-ключа". Буквально "все клубы всех стран всех видов спорта"
  статически захардкодить нечестно и невозможно без платного лицензированного
  провайдера данных (см. границу объёма в v2) — вместо этого сделано два
  реальных, честных шага: (1) страны теперь настоящий полный список, а не
  подмножество; (2) появился рабочий механизм *синхронизации* живых данных
  через API-ключ в структуру сущностей, а не просто демонстрация одного
  запроса, как было в `bukmeker connector` до этого.
- **`entities.py`: 249 реальных стран через `pycountry`** (официальный ISO
  3166-1, оффлайн-пакет, без сетевых вызовов и без риска ошибиться при ручном
  наборе) — это делает заявление "все страны мира" проверяемым фактом
  (`len(reg.countries) == 249`, тест на это есть), а не приближением из 5
  стран, как было раньше.
- **Лиги/клубы расширены до ~35 футбольных стран (реальные лиги и клубы,
  не вымышленные)**, плюс полный список всех 30 клубов NBA (это единственный
  случай, где "исчерпывающе" — правда: закрытая лига с фиксированным составом),
  плюс более широкий список игроков ATP. Это по-прежнему курируемая, а не
  исчерпывающая подборка — для >200 из 249 стран данных по лигам нет, и это
  явно показано в интерфейсе (см. ниже), а не скрыто.
- **Английский и шотландский футбол намеренно оба привязаны к стране GBR** —
  ISO 3166 не разделяет футбольные ассоциации внутри Великобритании, поэтому
  два разных `League` (Premier League и Scottish Premiership) с разными
  именами делят один `country_id`. Осознанный компромисс при сведении
  футбольного мира к ISO-справочнику стран, а не ошибка.
- **`bukmeker.connectors.sync.sync_registry_from_matches`** — превращает
  результат AI-коннектора (`CanonicalMatch` записи, полученные от любого
  провайдера по его API-ключу) в реальные `League`/`Competitor` записи в
  реестре сущностей. Это и есть "синхронизация всей игровой системы через
  API-ключ": не переписывание кода под каждого провайдера, а слияние того,
  что коннектор уже нормализовал, в единую схему сущностей.
- **Идемпотентность через явные ключи дедупликации**: лиги матчатся по
  `(sport_id, league_name)`, участники — по `(league_id, team_name)`. Повторный
  прогон синхронизации с теми же данными не создаёт дублей (проверено тестом
  `test_sync_is_idempotent_on_repeated_calls_with_same_matches`) — важно,
  потому что синхронизация по расписанию (например, ежедневная) — обычный
  сценарий использования, и без дедупликации реестр раздувался бы бесконечно.
- **`CanonicalMatch` не содержит страну** (большинство провайдеров спортивных
  данных не отдают её единообразно вместе с матчем), поэтому вновь
  обнаруженные лиги нельзя автоматически привязать к правильной реальной
  стране — вместо угадывания по названию команд, синхронизация явно требует
  `fallback_country_id` (флаг `--sync-country`, ISO alpha-3) и не притворяется,
  что определила страну надёжно. Отсутствующее имя лиги в источнике также не
  замалчивается: используется явная метка `"Unknown league (synced)"`, видимая
  как таковая, а не пустая строка.
- **`bukmeker connector --sync [--sync-sport ...] [--sync-country ...]`** —
  синхронизация лежит за тем же явным барьером реальных ключей, что и весь
  `bukmeker connector`: без `SOURCE_API_KEY`/`ANTHROPIC_API_KEY` команда
  по-прежнему отказывается работать (это не изменилось), синхронизация — лишь
  дополнительный шаг после реального fetch+normalize, а не отдельный обход
  этого требования.
- **Страница "🌍 Страны и лиги" дашборда переработана**: сверху — честные
  метрики (249 реальных стран vs. количество стран с фактическими данными по
  лигам), два режима просмотра ("по виду спорта" — как раньше, и "по стране" —
  поиск по всем 249 странам с явным сообщением "данных пока нет" для
  большинства из них вместо пустой таблицы или skip). Это прямое продолжение
  v5 (профессиональный, самообъясняющий интерфейс) — расширение охвата данных
  не должно создавать иллюзию полноты там, где её нет.
- **Тестовое покрытие нового пути** (`tests/test_sync.py` — 5 тестов на слияние/
  дедупликацию/пропуск неполных записей/заглушку лиги без имени;
  `tests/test_cli.py` — 3 интеграционных теста на `bukmeker connector --sync`,
  включая полный сквозной прогон с замоканными HTTP-запросом и конструктором
  Anthropic-клиента, без единого реального сетевого/AI вызова) — итого 166
  тестов, все проходят.

### Ключевые инженерные решения v9 (реальные имена команд, AI-коннектор в UI, современный купон)
- **Контекст решения.** Прямая обратная связь: "там названия команды нету" — на
  страницах Футбол/Баскетбол/Теннис не было ни одного реального имени, только
  абстрактные λ/коэффициенты; отдельно — просьба вынести ввод API-ключа и
  версии модели в дашборд (раньше это было только у `bukmeker connector` в
  CLI), и сделать "более современную генерацию купона с выбором вида спорта и
  чемпионата и дальнейшим анализом".
- **Внешние ссылки, данные откуда не использованы.** Пользователь прислал два
  URL: `https://etopaz.az/profile/my-account/personal-information` (страница
  личного кабинета за авторизацией — не запрашивалась и не могла быть открыта:
  это чужая частная учётная информация, доступа к которой ни у кого, кроме
  владельца сессии, быть не должно) и `https://www.gamebookers.com/en/sports`
  (публичная страница спортивных линий букмекера — попытка получить её через
  `WebFetch` дважды завершилась `ECONNRESET`, вероятно из-за защиты от ботов).
  Ни один реальный контент с этих сайтов не скопирован и не использован;
  реализация ниже основана на общеизвестном, не проприетарном паттерне UX
  (каскад "вид спорта → чемпионат → матч"), а не на копировании конкретного
  букмекера.
- **`_pick_match()`**: единая функция выбора лиги + двух участников из
  `EntityRegistry`, используемая и страницами прогнозов, и купоном. Лиги с
  <2 участниками исключаются из списка (нечего выбрать вторым соперником) —
  без этой проверки некоторые сид-лиги (например, Экстракласса с одним
  клубом) ломали бы выбор "В гостях".
- **Страницы Футбол/Баскетбол/Теннис теперь показывают реальные имена**
  (например, подзаголовок "Arsenal — Chelsea"), но λ/μ/коэффициенты
  по-прежнему вводятся вручную — честно, так как в seed-реестре нет
  исторических матчей этих команд для автоматического расчёта рейтинга
  (это как раз то, что дал бы AI-коннектор с реальными данными). Отображение
  результатов (`_probability_table`, `_render_value_section`) получило
  необязательный параметр `labels`, подменяющий технические ключи
  (`home_win`/`away_win`) на реальные имена только в выводе, не в расчёте.
- **Новая страница "🔌 AI-коннектор"** переносит `bukmeker connector`/`--sync`
  в браузер: поля URL/ключа провайдера, ключа и версии модели Anthropic
  (`claude-sonnet-5`/`claude-opus-4-8`/`claude-haiku-4-5-20251001`/
  `claude-fable-5`), опциональная синхронизация в реестр. Это единственная
  страница дашборда с реальными сетевыми и платными запросами — предупреждение
  об этом показано первым же элементом страницы, до всех полей ввода.
- **Редактирование секретов в сообщениях об ошибках (`_redact()`)**: если
  запрос к провайдеру или Anthropic падает, текст исключения (например,
  `requests`, где URL с query-параметром ключа попадает в сообщение об
  ошибке) очищается от введённых ключей перед показом в `st.error` — иначе
  ключ мог бы засветиться на экране/в истории браузера при простом сбое сети.
  Проверено тестом с реальным "утекающим" сообщением исключения.
- **"Современный" купон = гид с реальным анализом, а не просто более
  красивая форма.** Раздел "Собрать ставку через анализ" переиспользует ровно
  те же функции (`dixon_coles_matrix`/`basketball.predict_moneyline`/
  `tennis.predict_match_win_prob`), что и отдельные страницы спорта — после
  выбора вида спорта, чемпионата и команд/игроков вероятность считается, а не
  вводится руками, и сразу показан EV по каждому исходу перед добавлением в
  купон. Старый способ (ручной ввод строк таблицы) не удалён, а перенесён в
  сворачиваемый блок "для опытных" — экспресс-анализ и таблица объединяются
  в один список кандидатов перед генерацией купона.
- **`_stable_match_id()`**: один и тот же выбранный матч (лига + пара
  участников) всегда получает один и тот же `match_id` в рамках сессии —
  без этого два исхода одного реального матча (например, победа дома и
  ничья) выглядели бы для фильтра корреляции как независимые события, и
  система разрешила бы объединить их в один купон, хотя они взаимоисключающие.
- **Время матча (`st.datetime_input`)**: добавлено полем в гид анализа купона
  и сохраняется отдельно от `ValueBetCandidate` (в `st.session_state`, не в
  самом математическом dataclass'е) — время не участвует в расчёте
  EV/корреляции/Kelly, это чисто отображаемые метаданные, поэтому незачем
  раздувать ими core-класс. По той же причине честности, что и λ/коэффициенты:
  вводится вручную, так как в seed-реестре нет реальных расписаний матчей;
  подсказка у поля явно поясняет, что настоящее live-время появляется только
  через AI-коннектор с живым источником данных. Показывается и в таблице
  собранных ставок, и как "ближайший матч" купона в итоговом списке комбинаций.
- **Тестирование**: 6 новых тестов на реальные имена на страницах прогнозов;
  5 новых тестов на страницу AI-коннектора (включая мокнутый сквозной прогон
  fetch+sync и проверку редактирования секретов); 5 новых тестов на гид анализа
  купона (реальные команды в селекторах, вычисленный EV, добавление ноги,
  время матча, совместная генерация с ручными строками) — итого 184 тестов,
  все проходят.

### Ключевые инженерные решения v10 (статистика по купонам: день / месяц / всё время)
- **Контекст решения.** Прямая просьба: "промт для купона на один день, месяц
  и далее" — статистика/история по купонам, разбитая по периодам, а не
  только разовый расчёт одной выплаты.
- **`st.session_state["coupon_history"]`**: каждое нажатие "💾 Сохранить купон
  в историю" добавляет запись (время сохранения, время матча, названия ног,
  ставка, чистая прибыль) в список в session_state — история только в рамках
  текущей сессии браузера, без файла/БД (честно указано в подписи раздела;
  постоянное хранилище — уже отдельная тема, не часть текущего объёма).
- **Период считается от времени матча, если оно известно, иначе от времени
  сохранения** (`moment = record["match_time"] or record["saved_at"]`) —
  логично: "купоны на сегодня" должны означать "матчи сегодня", а не "когда я
  нажал кнопку"; для куда добавленных вручную (без времени матча) — честный
  fallback на момент сохранения, а не молчаливая ошибка.
- **Три периода (День/Месяц/Всё время)**, а не производный выбор дат — проще
  для пользователя и покрывает ровно то, что было запрошено буквально
  ("один день, месяц и далее"). Метрики за период: количество купонов, сумма
  ставок, чистая прибыль, ROI (`net_profit/stake`) — тот же набор метрик, что
  и в исходной спецификации `bukmeker.txt` (§2.3: ROI, Yield), без придумывания
  новых показателей.
- **Тестирование**: сохранение в историю и метрики за период по умолчанию;
  фильтрация по месяцу (запись, искусственно "состаренная" на 62 дня, не
  попадает в "Этот месяц", но видна во "Всё время"); очистка истории. Три
  новых теста, переключение периода — через `session_state`, а не
  `radio.set_value()` с текстовой меткой (та же особенность AppTest с
  `format_func`, что и раньше у `selectbox`) — итого 187 тестов, все проходят.

### Явные ограничения текущего объёма
- Node.js backend, Prisma/PostgreSQL/TimescaleDB, RabbitMQ, React-фронтенд, Docker
  Compose/Kubernetes и мониторинг (Prometheus/Grafana) из `bukmeker.txt` §1.2
  сознательно не реализованы — проект остаётся Python-библиотекой.
- **Монетизация — только расчёт.** `build_coupon_report` считает комиссию и
  чистую выплату; реальных платежей, счетов пользователей, эскроу или движения
  денег в проекте нет и не планируется в этом объёме — по прямому решению в
  этой сессии (альтернатива "реальные платежи Stripe/PayPal" требует юридической
  регистрации как букмекер/финсервис и была осознанно отклонена).
- **Список стран реальный и полный (249, ISO 3166-1 через `pycountry`), но
  лиги/клубы — курируемая, реальная, но не исчерпывающая подборка** (~35
  футбольных стран + вся NBA + топ ATP-игроки), не полная база каждого клуба
  каждой страны. Полное покрытие для конкретного провайдера достигается через
  `connectors/sync.py` и живые данные — в коде нет и не будет статичного
  списка тысяч клубов "на все случаи".
- **AI-коннектор тратит реальные деньги и требует реальной сети.** `bukmeker
  connector` и `ClaudeFieldMapper` при реальном использовании выполняют платный
  вызов Anthropic API и HTTP-запрос к стороннему провайдеру — это не бесплатная
  и не офлайн операция, в отличие от `bukmeker demo`.
- **Дашборд — локальный однопользовательский dev-сервер.** `bukmeker dashboard`
  запускает Streamlit без аутентификации, без multi-tenant изоляции и без
  production-развёртывания (reverse proxy, HTTPS, масштабирование) — подходит
  для локального использования и демонстраций, не для публичного хостинга.
