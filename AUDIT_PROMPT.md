# AUDIT & FIX PROMPT — Bukmeker (v7)

Полный аудит существующего проекта (математическое ядро, сущности, коннекторы,
дашборд, CLI) на корректность, безопасность, согласованность документации и
покрытие тестами — с обязательным исправлением найденного, а не только
перечислением. Выполняется автономно, без уточняющих вопросов пользователю.

## Роль
Ты — независимый ревьюер, который не писал этот код (даже если писал его в
предыдущей сессии) и проверяет его заново по каждому измерению ниже. Находка
без исправления и без теста, фиксирующего исправление, не считается закрытой.

## Измерения аудита

1. **Корректность математики** — перепроверить формулы в `features.py`,
   `ratings.py`, `models/goals.py`, `margin.py`, `calibration.py`,
   `value_betting.py`, `coupon.py`, `monetization.py`, `sports/*` на предмет
   пограничных случаев, не покрытых существующими тестами.
2. **Безопасность** — `auth.py` (сравнение пароля таймингово-безопасным
   способом), `connectors/*` (утечка ключей в логи/print, инъекции в URL),
   `dashboard.py` (не течёт ли пароль/ключи в интерфейс или session_state
   в открытом виде).
3. **Качество кода** — ruff чист, отсутствие мёртвого кода, консистентность
   сигнатур, отсутствие бессмысленных `except Exception: pass`.
4. **Покрытие тестами** — найти функции/ветки без прямого теста; особое
   внимание граничным условиям (пустые списки, нулевые/отрицательные входы,
   дублирующиеся id).
5. **Согласованность документации** — PROMPT.md/README.md/USAGE.md должны
   отражать актуальный код (число тестов, список команд, флагов CLI).
6. **Целостность данных** — `entities.py`: уникальность id, отсутствие дублей
   имён участников внутри одной лиги, консистентность seed-данных.

## Процесс
1. Прогнать `ruff check .` и `pytest tests/ -q` как базовую линию.
2. Пройти по каждому модулю целенаправленно (не по diff, а по всему файлу).
3. Каждую находку сформулировать как конкретный сценарий отказа (не общими
   словами "могло бы быть лучше").
4. Исправить, добавить/обновить тест, воспроизводящий сценарий из находки.
5. Повторный прогон `ruff` + `pytest` после каждой правки.
6. Обновить документацию, если правка меняет наблюдаемое поведение.
7. Зафиксировать в этом файле журнал находок и их статус (ниже).
